# Bug规范

### Bug等级

| 来源   | 等级          | 定义                                                         |
| ------ | ------------- | ------------------------------------------------------------ |
| 客户端 | 致命/非常严重 | 1、核心功能不可用，block测试：无法登录无法发消息功能无法展示功能无法打开2、主流程功能crash（主流程的case） 3、资损问题 |
|        | 严重          | 1、主流程功能bug（主流程的case）2、非主流程功能毕现crash（非主流程的 case）3、严重性能问题 |
|        | 一般          | 非主流程功能有问题（非主流程的 case）严重视觉交互或文案错误问题兼容性问题 |
|        | 提示/建议     | 1、体验建议2、基础视觉交互问题，不影响功能3、文案错误等，但不影响发版 |
| 服务端 | 致命/非常严重 | 1、核心功能不可用2、接口调用不成功3、服务挂掉4、资损问题     |
|        | 严重          | 1、主流程功能bug（主流程的case）2、性能问题3、安全问题4、影响到其他业务的主流程 |
|        | 一般          | 1、非主流程功能有问题（非主流程功能case）2、用户体感严重不佳的问题3、严重视觉交互或文案错误问题 |
|        | 提示/建议     | 1、文案问题，数据结构优化等不影响功能和发版的bug2、用户体感轻微不佳的问题 |



### Bug优先级

强调 紧急/高优Bug须当日修复，中级Bug须T+1内修复(特殊可议)

Bug优先级分为四个等级：

- 紧急：阻塞主流程，相关开发人员立即进行修复，一般多指严重block测试进度和系统不可用BUG。
- 高：影响业务，当天需要进行修复，一般指Block部分功能模块测试和使用的BUG

- 中：不一定马上修改，但需发版前必须修正；一般指常规BUG，不构成系统阻塞和发布风险
- 低：如果时间允许应该修改，时间不够的话允许暂不修改；一般为建议性BUG，不影响产品功能和属性的使用。（例如log打印、用户体感轻微不佳的问题等）



### Bug状态定义

| **分类**   | **状态**                        | **关注者** |
| ---------- | ------------------------------- | ---------- |
| **待解决** | **新****接受/处理****重新打开** | 研发       |
| **已解决** | **已解决****已验证****已拒绝**  | 质量       |
| **已关闭** | **已关闭**                      |            |

待解决：

**新：**提交的新bug。

**重新打开**：已解决的bug验证问题仍然存在，需及时置为Reopen，并备注相应的验证版本。

已解决：

**已解决**：开发解决后需要及时将bug状态置为Fixed状态，并认真填写bug产生原因；

**已验证**：已解决的bug验证问题未重现，必现性的可立即关闭，概率性的需验证至少三个版本方可关闭，并备注相应的验证版本。测试在验证bug的时候，需要和开发确认下bug原因和解决方法，check问题定位原因。

**已拒绝**：1、确认问题存在（与需求不一致），综合考虑修复成本，项目时间，对客影响等因素，产研测达成一致。 由需求人员置为later。

2、提交的bug单重复了（操作步骤和问题原因都一致），拒绝时需要添加重复的bug单ID

3、提交的bug不是问题，测试开发沟通确认达成一致，可置为已拒绝，并添加相关原因。否则容易产生reopen。

已关闭：

**已关闭**：已解决/已验证状态下，通过验证可置为已关闭状态，已拒绝状态经过产品&开发&测试多方达成一致后确认无需处理或者为非BUG问题，可关闭
